# Email Templating: Evaluation & Recommendation

## Current State

- **Engine**: Fluid.Core 2.22.0 (Liquid template engine for .NET)
- **Files**: 5 `.scriban` files embedded as assembly resources in `CoreBanking.Infrastructure`
- **Syntax**: Simple `{{PropertyName}}` interpolation only — no conditionals, loops, or complex logic
- **Loading**: `Assembly.GetManifestResourceStream` with `ConcurrentDictionary` cache

## Options Evaluated

### Option A: Keep Fluid.Core (Recommended)
- **Pros**: Actively maintained, fast, lightweight, works perfectly with .NET 10, handles current templates trivially
- **Cons**: Liquid syntax is limited (but we don't need more)
- **Verdict**: Right tool for the job. Templates only use `{{PropertyName}}` — Liquid is more than sufficient.

### Option B: Switch to RazorLight (cshtml)
- **Pros**: Razor syntax, familiar to .NET devs, strongly typed `@Model` support
- **Cons**: **Abandoned** (last NuGet update 2020, v2.3.1), no .NET 8+ support guaranteed, runtime compilation overhead, heavier dependency, requires `PreserveCompilationContext`
- **Verdict**: Not recommended. The package is dead and will become a liability.

### Option C: Switch to actual Scriban
- **Pros**: More powerful than Liquid (`{{- -}}` trimming, `func`, `include`, loops), actively maintained
- **Cons**: Heavier than Fluid for what we need, different package
- **Verdict**: Overkill. We don't use any Scriban-specific features.

### Option D: Use Microsoft.AspNetCore.Razor built-in
- **Pros**: First-party, always maintained
- **Cons**: Complex setup for email templates (need Razor views infrastructure, `IServiceProvider`, etc.), overkill for static HTML with property interpolation
- **Verdict**: Too much machinery for simple email templates.

## Recommendation

**Keep Fluid.Core** but clean up the naming confusion:

1. Rename `.scriban` → `.liquid` (matches the actual engine — Fluid is a Liquid port)
2. Remove `PreserveCompilationContext=true` from both `.csproj` files (leftover from RazorLight, not needed)
3. Update `EmailService.cs` resource name pattern from `.scriban` to `.liquid`
4. Add missing templates: `PasswordResetEmail.liquid`, `LoanApprovedEmail.liquid` (model classes exist but templates don't)
