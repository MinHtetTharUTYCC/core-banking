# Plan: Pagination + Sorting Refactor

## Context
Sorting was added to transaction queries but implemented in-memory in the handler (handler gets all rows, then sorts in C#). This is a learning project for enterprise patterns. Best practice is to push sorting + pagination down to the **repository/EF Core level** so the database does the heavy lifting (`ORDER BY` + `SKIP/TAKE` at SQL level).

## Best Practice Pattern

```
Controller ──[FromQuery] sortBy, page, pageSize──► Query
    ──► Handler ──► Repository(sortBy, page, pageSize)
        ──► EF Core: WHERE + ORDER BY + SKIP/TAKE + COUNT
            ──► PaginatedResult<T> (Items, TotalCount, Page, PageSize, TotalPages)
```

Key principles:
- **Reusable `PaginatedResult<T>`** — shared DTO in `Application/Common/`
- **Database-level sorting + paging** — `IQueryable` chain before `ToListAsync()`
- **TotalCount** for client-side pagination UI
- **Defaults** — page=1, pageSize=20, sortBy=Newest
- **Sort via enum** — avoids injection, maps to `IQueryable.OrderBy` in repository

## Implementation Steps

### 1. Create `PaginatedResult<T>` (reusable)
**Create:** `src/CoreBanking.Application/Common/Models/PaginatedResult.cs`

```csharp
public class PaginatedResult<T>
{
    public List<T> Items { get; init; }
    public int TotalCount { get; init; }
    public int Page { get; init; }
    public int PageSize { get; init; }
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
    public bool HasPreviousPage => Page > 1;
    public bool HasNextPage => Page < TotalPages;
}
```
